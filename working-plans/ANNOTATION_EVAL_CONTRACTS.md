# Annotation And Evaluation Contracts

This document defines the data contracts between:

- candidate extraction
- annotation review
- benchmark export
- offline evaluation

The goal is to keep the system modular. Detection, review UI, sampling, and evaluation should be able to evolve without inventing incompatible formats.

Additional architectural requirement:

- the review workflow must run fully locally against locally accessible images
- finalized benchmark datasets must be exportable as standalone image assets plus manifests so they remain usable independently of the Lightroom catalog

## Contract Boundaries

The pipeline is split into four handoff points:

1. extractor output
2. annotation record
3. benchmark manifest row
4. evaluation result row

These should remain distinct.

Reason:

- extractor output is not truth
- annotation records contain workflow metadata that should not always leak into benchmark manifests
- benchmark manifests should be portable and stable
- evaluation results should be append-only outputs, not edits to source manifests

## General Format Rules

- JSONL is the canonical interchange format for exported rows
- SQLite is the canonical local persistence layer for the review tool
- all timestamps should be ISO-8601 strings in UTC when generated by tooling
- all filesystem paths in exported contracts should be absolute paths
- booleans should be explicit `true`/`false`, not omitted to imply false

## Canonical Identity Rules

There are four distinct identity concepts:

- `candidate_id`: one detector-produced review target
- `sample_id`: one benchmark sample row
- `source_image_id`: original Lightroom image identifier if known
- `truth_label`: canonical taxonomy label used for scoring

Do not reuse one field for another purpose.

### `truth_label`

`truth_label` should be the canonical scoring identity.

Recommendation:

- store both `truth_label` and `truth_sci_name`
- allow `truth_common_name` for review ergonomics
- score against `truth_label`

Reason:

- common names are reviewer-friendly but not stable enough for scoring
- scientific names are human-readable but can drift with taxonomy changes
- canonical labels are the best scoring key if they are normalized consistently

## Extractor Output Contract

Extractor output represents candidate bird detections before human validation.

One row corresponds to one detection candidate.

Required fields:

```json
{
  "candidate_id": "cand_000001",
  "source_image_id": 123456,
  "source_image_path": "/abs/path/to/image.ARW",
  "preview_image_path": "/abs/path/to/previews/cand_000001.jpg",
  "detector_name": "yolo-bird-v1",
  "detected_class": "bird",
  "detector_confidence": 0.93,
  "bbox": {
    "x1": 1120,
    "y1": 540,
    "x2": 1680,
    "y2": 1210
  },
  "bbox_area_fraction": 0.08,
  "capture_datetime": "2024-05-11T08:41:22Z",
  "review_status": "unreviewed"
}
```

Recommended optional fields:

```json
{
  "folder": "/abs/path/to/folder",
  "rating": 3.0,
  "gps_lat": 39.1234,
  "gps_lon": -76.5432,
  "region_hint": "us_northeast",
  "lens_model": "FE 200-600mm F5.6-6.3 G OSS",
  "focal_length": 600.0,
  "existing_keywords": ["Birds|Order|Hawks-Eagles-Kites-Allies"],
  "burst_group_id": "burst_20240511_084122_001",
  "safe_single_subject_burst": true,
  "near_duplicate_group": "dup_20240511_084122_a"
}
```

Notes:

- `preview_image_path` is the preview used by the review UI
- the default v1 preview is the full image with detection box overlay
- crop views, if used, should be derived views, not required exported assets
- extractor output must not claim truth labels

## Annotation Record Contract

Annotation records represent human-reviewed decisions for a candidate.

One row corresponds to one reviewed candidate.

Required fields for any saved annotation:

```json
{
  "candidate_id": "cand_000001",
  "annotation_status": "labeled",
  "stress": false,
  "reject": false,
  "unsure": false,
  "not_a_bird": false,
  "bad_crop": false,
  "duplicate": false,
  "annotated_at": "2026-04-22T16:05:00Z"
}
```

Required additional fields for a labeled positive sample:

```json
{
  "truth_common_name": "Bald Eagle",
  "truth_sci_name": "Haliaeetus leucocephalus",
  "truth_label": "00419_Animalia_Chordata_Aves_Accipitriformes_Accipitridae_Haliaeetus_leucocephalus",
  "taxon_class": "Aves",
  "resolved_from_input": "bald eagle"
}
```

Recommended optional fields:

```json
{
  "notes": "",
  "annotated_by": "local-user"
}
```

### Allowed `annotation_status` Values

- `labeled`
- `reject`
- `unsure`
- `not_a_bird`
- `bad_crop`
- `duplicate`
- `skipped`

Rules:

- `stress=true` is only valid when `annotation_status='labeled'`
- `truth_*` fields are required for `labeled`
- `truth_*` fields should normally be absent for `not_a_bird`
- `duplicate` may keep `truth_*` fields if the reviewer labeled it before marking duplicate

## Benchmark Manifest Contract

Benchmark manifests are portable evaluation inputs. They should contain truth and enough context for slicing metrics, but should avoid review-tool-only implementation details.

One row corresponds to one benchmark sample.

Required fields:

```json
{
  "sample_id": "catalog-realworld-000001",
  "source_type": "catalog_real_world",
  "image_path": "/abs/path/to/image.ARW",
  "truth_label": "00419_Animalia_Chordata_Aves_Accipitriformes_Accipitridae_Haliaeetus_leucocephalus",
  "truth_sci_name": "Haliaeetus leucocephalus",
  "truth_common_name": "Bald Eagle",
  "taxon_class": "Aves",
  "multi_label": false,
  "secondary_truth_labels": []
}
```

Recommended optional fields:

```json
{
  "source_image_id": 123456,
  "candidate_id": "cand_000001",
  "region": "us_northeast",
  "country_code": "US",
  "state_code": "MD",
  "capture_datetime": "2024-05-11T08:41:22Z",
  "difficulty": "medium",
  "stress": false,
  "reject": false,
  "detector_name": "yolo-bird-v1",
  "detector_confidence": 0.93,
  "bbox": {
    "x1": 1120,
    "y1": 540,
    "x2": 1680,
    "y2": 1210
  },
  "notes": ""
}
```

Optional exported-dataset fields:

```json
{
  "exported_image_path": "/abs/path/to/exported_dataset/images/catalog-realworld-000001.jpg",
  "export_format": "jpeg",
  "export_width": 2048,
  "export_height": 1365
}
```

### `source_type`

Allowed values:

- `control_inat`
- `catalog_real_world`
- `catalog_stress`
- `catalog_reject`

Reason:

- benchmark outputs should preserve provenance
- metrics should be reported separately by source type

### Multi-Label Fields

V1 scoring is single-label-first, but the manifest should reserve shape for future multi-label use.

Rules:

- `multi_label` is always present
- `secondary_truth_labels` is always present
- when `multi_label=false`, `secondary_truth_labels` should be `[]`
- `truth_label` remains the primary label

This is deliberately conservative. It preserves forward compatibility without forcing v1 to solve detection-plus-classification scoring.

## Benchmark Export Rules

These rules convert annotation records into benchmark manifests.

There are two export targets:

- manifest-only export, which preserves references back to source images
- materialized dataset export, which writes standalone JPEG assets and manifests to a durable dataset directory

Recommendation:

- materialized dataset export should be the default for finalized benchmark sets

Reason:

- it decouples evaluation from Lightroom catalog availability
- it makes the benchmark reproducible and portable
- it protects against preview-cache churn or catalog reorganization

### Export To `catalog_real_world`

Include only candidates where:

- `annotation_status='labeled'`
- `stress=false`
- `reject=false`
- `unsure=false`
- `not_a_bird=false`
- `bad_crop=false`
- `duplicate=false`

### Export To `catalog_stress`

Include only candidates where:

- `annotation_status='labeled'`
- `stress=true`

### Export To `catalog_reject`

Include candidates where any of the following is true:

- `annotation_status='reject'`
- `annotation_status='not_a_bird'`
- `reject=true`
- `not_a_bird=true`

Open question:

- whether `bad_crop` and `unsure` should always export into `catalog_reject` or remain a separate quarantine pool

Recommendation:

- do not export `unsure` into benchmark manifests by default
- do not export `bad_crop` by default
- keep both available for manual inclusion later

## Materialized Dataset Export Contract

When exporting a finalized benchmark set, each sample should be rendered or copied to a durable standalone image file.

Requirements:

- output images should be JPEG
- output paths should be stable and manifest-addressable
- manifests should record both source image path and exported dataset image path when available
- exported datasets should be runnable without Lightroom catalog access
- exported JPEGs should obey a configured maximum file-size cap
- exported JPEGs should not be compressed below a configured minimum quality floor

Open implementation question:

- whether the exported JPEG should come from the best available fast preview source or from a higher-fidelity render path
- what the default maximum exported JPEG size should be

Recommendation:

- prefer a deterministic export path suitable for reproducible evaluation once the benchmark has been finalized
- do not rely on transient review-preview assets as the benchmark artifact
- enforce file-size caps by export policy rather than leaving size unconstrained
- treat the file-size cap as the primary export constraint, with a quality floor as a safeguard against pathological over-compression

Recommended manifest fields for materialized exports:

```json
{
  "exported_image_size_bytes": 482193,
  "export_max_size_bytes": 1048576,
  "export_quality": 85,
  "export_quality_floor": 75
}
```

## Evaluation Input Contract

The evaluation runner should accept one or more manifest files plus explicit run configuration.

Recommended config shape:

```json
{
  "manifest_paths": [
    "/abs/path/to/catalog_real_world.jsonl"
  ],
  "models": ["general", "birds"],
  "ensemble": "none",
  "top_k": 5,
  "min_confidence": 0.25,
  "geo_filter": true,
  "region_source": "manifest"
}
```

Important rule:

- evaluation config belongs to the run, not embedded per sample unless sample-specific behavior is required

## Evaluation Result Contract

Evaluation output should be append-only and should never mutate source manifests.

Each per-sample result row should contain:

```json
{
  "run_id": "eval_2026-04-22T16-30-00Z",
  "sample_id": "catalog-realworld-000001",
  "source_type": "catalog_real_world",
  "image_path": "/abs/path/to/image.ARW",
  "truth_label": "00419_Animalia_Chordata_Aves_Accipitriformes_Accipitridae_Haliaeetus_leucocephalus",
  "truth_sci_name": "Haliaeetus leucocephalus",
  "truth_common_name": "Bald Eagle",
  "taxon_class": "Aves",
  "models": {
    "birds": [
      {
        "label": "00419_Animalia_Chordata_Aves_Accipitriformes_Accipitridae_Haliaeetus_leucocephalus",
        "score": 0.88,
        "rank": 1
      }
    ]
  },
  "ensemble": [],
  "final_predictions": [
    {
      "label": "00419_Animalia_Chordata_Aves_Accipitriformes_Accipitridae_Haliaeetus_leucocephalus",
      "score": 0.88,
      "rank": 1
    }
  ],
  "correct_top1": true,
  "correct_top3": true,
  "accepted": true,
  "runtime_ms": 183
}
```

Recommended optional fields:

```json
{
  "geo_filter_applied": true,
  "region_used": "us_northeast",
  "abstained": false,
  "error": null
}
```

## Summary Output Contract

Each evaluation run should emit a summary file alongside per-sample outputs.

Required top-level fields:

```json
{
  "run_id": "eval_2026-04-22T16-30-00Z",
  "created_at": "2026-04-22T16:30:00Z",
  "config": {},
  "metrics": {
    "top1_accuracy": 0.82,
    "top3_accuracy": 0.91,
    "accepted_precision": 0.89,
    "coverage": 0.76,
    "mean_runtime_ms": 183.2
  },
  "slices": {
    "source_type": {},
    "taxon_class": {},
    "region": {}
  }
}
```

Recommended slice breakdowns:

- by `source_type`
- by `taxon_class`
- by `region`
- by `difficulty` when present

## Contract Validation Rules

Validation should happen at each boundary.

Minimum validation requirements:

- extractor rows must have valid bbox coordinates and preview path
- labeled annotations must have all `truth_*` fields
- benchmark manifests must have valid `source_type`
- benchmark manifests must always include `multi_label` and `secondary_truth_labels`
- evaluation result rows must preserve the input `sample_id`

If a row is invalid, the tool should fail loudly rather than silently producing a malformed downstream file.

## Deliberate Omissions

These are intentionally not part of the stable contract yet:

- exact taxonomy alias resolution format
- calibration tables for model-score normalization
- burst-level batch annotation provenance
- crop-image export format

Reason:

- these are implementation details that may change before the core workflow stabilizes

## Recommended Next Step

Once these contracts are accepted, the next implementation-facing specs should be:

- SQLite schema mapped to these contracts
- review state transitions
- manifest validator behavior
